# SCDC 数据搜索与清洗方案 - 架构分析

## 一、系统概览

SCDC 采用 **四阶段 Agent 流水线** 架构，完成从数据搜索到报告生成的全自动化流程。

### 流水线流程图
```
用户输入主题
    ↓
OrchestratorAgent（主控）
    ↓
┌──────────────────────────────────────────────────┐
│  CollectorAgent  →  CleanerAgent  →  AnalyzerAgent  →  ReporterAgent  │
│  （采集）           （清洗）          （分析）         （报告）       │
──────────────────────────────────────────────────┘
    ↓
返回结构化报告
```

---

## 二、各 Agent 职责与实现

### 1. CollectorAgent（数据采集 Agent）

**职责**：搜索互联网信息，获取原始数据。

**工作流程**：
1. 调用 `SerpAPIService` 对主题关键词执行搜索（默认取前 `max_items` 条结果）
2. 并发（`asyncio.gather`）爬取每个搜索结果的目标页面全文
3. 爬取失败时降级使用搜索引擎返回的摘要（snippet）

**使用的底层组件**：
- `SerpAPIService`：Google 搜索 API 服务
- `HTTPCrawler`：HTTP 网页爬虫，支持指数退避重试、User-Agent 轮换

**输出**：`CollectedItem` 列表，包含 source_type（search_crawl / search_snippet）、source_uri、title、content、metadata

---

### 2. HTMLCleaner（HTML 内容清洗器，非 Agent，被 HTTPCrawler 使用）

**职责**：从原始 HTML 中提取标题、纯文本内容和元数据。

**清洗规则**：
- 提取 `<title>` 标签
- 提取 `<meta>` 标签中的 description/keywords 等元数据
- 彻底删除噪声标签：`script`、`style`、`nav`、`footer`、`header`、`iframe`、`noscript`
- 合并空行，返回干净纯文本

**为什么清洗**：
- 降低 LLM token 成本（去除 JS/CSS/导航噪声）
- 纯文本更便于 CleanerAgent 做语义处理

---

### 3. CleanerAgent（数据清洗 Agent）

**职责**：接收原始采集数据，执行三道清洗工序，提升数据质量。

**清洗规则**（严格执行）：
1. ** 短内容过滤**：丢弃内容长度 < 20 字符的条目（可配置）
2. **② URI 去重**：相同 URL 只保留首次出现
3. **③ 内容指纹去重**：不同 URL 但内容相同的条目，通过 MD5 哈希指纹识别并排除
4. **④ 文本分块**：将长文本按 1000 字符分块，在换行处断句保持语义完整

**输出**：`CleanedItem` 列表，包含 source_type、source_uri、title、summary（前 200 字符摘要）、content_chunks（分块列表）、relevance_score、metadata

---

### 4. AnalyzerAgent（AI 分析 Agent）

**职责**：接收清洗后的数据，调用 LLM 进行深度分析，提取洞察。

**工作流程**：
1. 从 Milvus 向量库检索与主题相关的历史上下文（最多 3 条）
2. 拼接清洗后数据 + 历史上下文，构建结构化 Prompt
3. 调用 LLM（gpustack / ollama），要求返回 JSON 格式分析结果
4. 验证并规范化 LLM 返回的维度、证据 URI、置信度

**降级策略**：LLM 不可用时，自动降级为基于规则的分析（直接提取原文摘要）

**输出**：`AnalyzerOutput`，包含 summary（执行摘要）、insights（洞察列表，每条包含结论、分析、证据、置信度、维度）

---

### 5. ReporterAgent（报告生成 Agent）

**职责**：接收 AnalyzerAgent 的分析结果，调用 LLM 撰写完整叙事性报告。

**工作流程**：
1. 按分析维度对洞察进行分组
2. 构建专业市场分析报告 Prompt，要求 LLM 撰写董事会级别报告
3. 调用 LLM 生成 1500-3000 字中文报告
4. 生成 ECharts 饼图配置（维度分布）和柱状图配置（各维度置信度分布）
5. 使用 matplotlib 渲染图表为 base64 PNG 图片
6. 拼接报告头、LLM 报告内容、来源引用、图表图片

**降级策略**：LLM 不可用时，使用模板组装模式（结构化拼接洞察列表）

**输出**：`ReporterOutput`，包含 markdown_report、chart_configs（ECharts 配置）、chart_images（base64 图片）

---

### 6. OrchestratorAgent（主控调度 Agent）

**职责**：串联四阶段流水线，管理状态回调。

**工作流程**：
1. queued → collecting → cleaning → analyzing → reporting → completed/failed
2. 每个阶段通过 `state_callback` 回调通知外部系统更新任务状态
3. 任一阶段失败立即终止，返回部分结果

---

## 三、当前方案的质量优势

| 环节 | 质量保证措施 |
|------|-------------|
| **搜索** | SerpAPI（Google 搜索）提供高质量搜索结果，非普通爬虫 |
| **爬取** | 指数退避重试 + User-Agent 轮换 + 10 秒超时控制 |
| **HTML 清洗** | 彻底去除 JS/CSS/导航噪声，提取纯正文 |
| **内容去重** | URI 去重 + MD5 内容指纹去重，避免重复分析 |
| **短内容过滤** | 丢弃 < 20 字符的低质量条目 |
| **文本分块** | 1000 字符分块，换行处断句保持语义完整性 |
| **LLM 降级** | LLM 不可用时自动降级为规则分析，不中断流程 |
| **历史上下文** | 向量库检索历史相关报告，支持跨期对比分析 |
| **证据追溯** | 每条洞察附带证据 URI + 置信度评分 |

## 四、可优化方向建议

### 4.1 搜索阶段优化

| 优化点 | 建议 | 预期效果 |
|-------|------|---------|
| **多搜索引擎融合** | 当前仅使用 SerpAPI，可增加 Bing API、本地 Elasticsearch 等作为备用 | 搜索结果覆盖面提升，避免单一引擎偏差 |
| **搜索参数精细化** | 当前仅传 query，可增加 time_range、site 限定、语言筛选等参数 | 搜索结果更精准 |
| **搜索质量过滤** | 对搜索结果做初步评分，优先爬取高相关性页面 | 减少无效爬取，降低成本 |

### 4.2 清洗阶段优化

| 优化点 | 建议 | 预期效果 |
|-------|------|---------|
| **动态最小长度阈值** | 当前硬编码 20 字符，可根据行业调整 | 更适配不同领域的数据特征 |
| **语义去重** | 当前 MD5 指纹只能精确去重，可增加基于 Embedding 的相似度去重 | 识别语义相近但文字不同的内容 |
| **语言识别** | 过滤非目标语言内容 | 避免噪声干扰 LLM 分析 |
| **质量评分** | 引入基于规则的页面质量评分（如文本/HTML 比例、段落数等） | 优先保留高质量内容 |

### 4.3 分析阶段优化

| 优化点 | 建议 | 预期效果 |
|-------|------|---------|
| **Rerank 排序** | 当前已有 Rerank 模型配置，但在 AnalyzerAgent 中未使用。可在向量库检索后对候选文档用 Rerank 模型做精准排序 | 提升上下文相关度，让 LLM 看到最相关的材料 |
| **增量上下文窗口管理** | 当前硬编码截断 1500 字符，可根据 LLM 的 max_tokens 动态分配上下文 | 最大化利用 LLM 上下文窗口 |
| **多轮分析** | 当前单次 LLM 调用，可增加"初筛 → 深析"两阶段流程 | 先做广度扫描再做深度挖掘 |
| **置信度校准** | LLM 给出的置信度可能不准，可基于证据数量、来源质量做后验校准 | 更可靠的置信度评估 |

### 4.4 报告阶段优化

| 优化点 | 建议 | 预期效果 |
|-------|------|---------|
| **配图生成** | 当前已有 `image_generation.py`（图像生成服务），可在报告中自动插入配图 | 提升报告可读性和专业感 |
| **报告模板多样化** | 当前固定结构，可增加多种模板（如竞品分析模板、技术趋势模板等） | 更贴合不同场景需求 |
| **关键指标提取** | 自动提取报告中的关键数字/趋势，生成结构化指标卡 | 便于快速阅读和对比 |

### 4.5 整体流程优化

| 优化点 | 建议 | 预期效果 |
|-------|------|---------|
| **数据源管理增强** | 当前数据源管理在 `data_sources.py`，可增加自动发现、健康检查、优先级排序等功能 | 更智能的数据源调度 |
| **缓存机制** | 对搜索结果、爬取内容、LLM 分析结果增加 Redis 缓存 | 降低重复分析成本，提升响应速度 |
| **流水线可配置** | 当前流程写死，可支持通过配置跳过某些阶段或调整参数 | 更灵活的场景适配 |

## 五、Rerank 模型集成方案

当前系统已配置了 Rerank 模型（`lb-reranker-0.5b-v1.0-gguf`），但尚未在流水线中使用。

### 推荐集成位置：AnalyzerAgent 的历史上下文检索

**当前流程**：
```
主题关键词 → Embedding → 向量库相似搜索 → 取 Top 3 → 直接喂入 LLM
```

**优化流程**：
```
主题关键词 → Embedding → 向量库相似搜索（召回 Top 20） → Rerank 模型重排序 → 取 Top 3 → 喂入 LLM
```

### 实现价值：
1. **先召回后排序**：向量搜索擅长广撒网（高召回率），Rerank 擅长精准排序（高精确率）
2. **上下文质量提升**：让 LLM 看到与主题最相关的历史报告片段，而不是仅仅向量相似度高的片段
3. **成本可控**：Rerank 模型轻量，排序 20 个候选延迟通常 < 100ms

## 六、总结

| 维度 | 当前方案 | 评价 |
|-----|---------|------|
| **搜索能力** | SerpAPI + HTTP 爬取 | ✅ 成熟可靠 |
| **HTML 清洗** | BeautifulSoup 去噪提取 | ✅ 有效降低噪声 |
| **数据去重** | URI + MD5 内容指纹 | ✅ 精确去重，缺语义去重 |
| **LLM 分析** | gpustack/ollama + 降级策略 | ✅ 健壮性好 |
| **报告生成** | LLM 撰写 + 图表渲染 | ✅ 专业级别 |
| **流水线编排** | 四阶段串联 + 状态回调 | ✅ 可观测性好 |
| **Rerank 集成** | 已配置但未使用 | ⚠️ 待集成 |
| **语义去重** | 不支持 | ️ 可增强 |
| **多源搜索** | 仅 SerpAPI | ⚠️ 可扩展 |

整体方案质量较高，建议优先集成 **Rerank 模型** 和 **语义去重**，这两项可带来显著的质量提升。
