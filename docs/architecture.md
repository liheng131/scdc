## 1. 文档定位与维护边界

### 1.1 本文档回答的问题

本文件回答以下问题：

- 系统由哪些运行时模块组成
- 模块之间如何协作
- 数据、任务、接口、通知的边界在哪里
- 哪些是必须遵守的架构约束

本文件**不负责**记录具体依赖版本、安装命令、许可证清单，也不承担任务排期职责。

### 1.2 多 Agent 使用约定

| Agent | 使用方式 | 允许更新的内容 |
|------|------|------|
| 设计 Agent | 产出可执行技术方案时以本文为主 | 模块边界、流程、接口契约、决策记录 |
| 开发 Agent | 实现前先核对是否符合本文约束 | 当实现被确认优于旧方案时，可回写实现后架构 |
| 审阅 Agent | 审查实现是否偏离架构约束 | 回写架构偏差、遗漏约束、需要补充的契约 |

### 1.3 文档边界

| 内容 | 应写在哪个文档 |
|------|------|
| 业务目标、验收标准 | `prd.md` |
| 模块职责、数据流、接口边界 | `architecture.md` |
| 依赖选型、版本、目录规范、命令 | `tech.md` |
| 任务拆解、状态、协作闭环 | `plan.md` |

## 2. 总体架构

### 2.1 架构原则

| 原则 | 说明 |
|------|------|
| 前后端分离 | 前端通过 REST API 与 SSE 消费后端能力 |
| 模块化单体优先 | MVP 阶段采用 Modular Monolith，保留后续拆分能力 |
| 异步优先 | 采集、分析、导出、通知等长耗时任务均走异步队列 |
| Agent 职责单一 | 每个 Agent 只负责一个清晰业务阶段 |
| 状态可追踪 | 任务状态、Agent 状态、导出状态必须可观察 |
| 证据可追溯 | 报告结论必须能回溯到原始来源或处理中间结果 |

### 2.2 逻辑分层

| 层级 | 核心职责 | 主要模块 |
|------|------|------|
| 表现层 | 用户交互、结果展示、状态反馈 | Web UI、Dashboard、Chat、Reports |
| 接入层 | 鉴权、路由、参数校验、统一响应 | API、SSE、Auth |
| 编排层 | 任务编排、状态机、重试与超时控制 | Orchestrator、Task Manager |
| 执行层 | 采集、清洗、分析、导出、通知 | Workers、Agents、Parsers、Exporters |
| 数据层 | 结构化数据、向量数据、对象存储、缓存 | PostgreSQL、Milvus、MinIO、Redis、OpenSearch |

### 2.3 模块职责边界

| 模块 | 负责 | 不负责 |
|------|------|------|
| 前端应用 | 发起请求、展示状态、渲染结果、导出入口 | 业务规则判定、采集逻辑 |
| API 服务 | 用户请求入口、权限校验、任务创建、结果查询 | 长耗时任务执行 |
| 调度 Agent | 流程编排、状态流转、失败重试、任务收敛 | 直接生成业务洞察 |
| 采集 Agent | 搜索、爬取、拉取原始数据 | 数据标准化、报告产出 |
| 清洗 Agent | 去重、标准化、实体抽取、翻译 | 业务判断与结论生成 |
| 分析 Agent | 生成结论、证据、置信度 | 报告排版、通知发送 |
| 报告 Agent | 将分析结果结构化为可交付内容 | 重新定义分析结论 |
| 通知模块 | 邮件/IM 推送 | 生成报告内容 |

### 2.4 运行时拓扑

| 组件 | 运行形态 | 作用 |
|------|------|------|
| Nginx | 网关容器 | 静态资源、HTTPS、反向代理 |
| Frontend | 静态站点 | Web 界面 |
| FastAPI | API 容器 | 同步接口、SSE、任务入口 |
| Celery Workers | 工作容器 | 长耗时任务执行 |
| Celery Beat | 调度容器 | 定时任务触发 |
| PostgreSQL | 数据库 | 结构化业务数据 |
| Redis | 缓存/队列 | Broker、状态缓存、限流 |
| OpenSearch | 检索引擎 | 全文检索、聚合 |
| Milvus | 向量库 | 语义检索、历史知识复用 |
| MinIO | 对象存储 | 文档、导出报告、附件 |
| SearXNG | 搜索聚合 | 搜索型采集能力 |
| Ollama | 模型推理服务 | 本地 LLM 推理 |

## 3. 数据与状态边界

### 3.1 存储职责分配

| 存储 | 存什么 | 为什么放这里 |
|------|------|------|
| PostgreSQL | 用户、角色、数据源、任务、报告元数据、模板、通知配置 | 强事务、结构化查询 |
| Redis | 会话、短期缓存、任务中间状态、队列消息 | 低延迟、临时性强 |
| OpenSearch | 全文索引、检索索引、聚合结果 | 面向搜索与分析 |
| Milvus | 向量索引、历史知识片段 | 语义检索与相似召回 |
| MinIO | 上传文档、导出文件、原始附件 | 面向二进制对象存储 |

### 3.2 核心实体

| 实体 | 关键字段 | 说明 |
|------|------|------|
| User | id, role, status | 系统用户 |
| DataSource | type, config, schedule, enabled | 数据源配置 |
| Task | type, trigger_mode, status, input, output_ref | 分析任务 |
| TaskRun | task_id, stage, started_at, ended_at, result | 单次执行记录 |
| Report | task_id, version, status, storage_ref | 报告产物 |
| Template | scope, version, content, status | 报告模板 |
| NotificationRule | channel, trigger, target, enabled | 通知规则 |

### 3.3 任务状态机

| 状态 | 含义 |
|------|------|
| `created` | 任务已创建，尚未排队 |
| `queued` | 已进入队列等待执行 |
| `collecting` | 正在采集 |
| `cleaning` | 正在清洗 |
| `analyzing` | 正在分析 |
| `reporting` | 正在生成报告 |
| `notifying` | 正在发送通知 |
| `completed` | 执行完成且产物可用 |
| `failed` | 当前执行失败 |
| `cancelled` | 用户或系统取消 |

### 3.4 证据追溯约束

- 每个核心结论必须能关联至少一个来源证据
- 报告中的图表必须能追溯到结构化数据集
- 上传文档、爬虫结果、搜索结果都需要保留来源标识
- 审阅阶段若无法完成追溯，应直接判定任务未完成

## 4. 核心流程

### 4.1 问答触发流程

1. 用户在前端发起自然语言问题。
2. API 完成鉴权、生成任务记录，并返回会话标识。
3. 调度 Agent 创建执行图，分派采集、清洗、分析、报告阶段。
4. 阶段进度通过 SSE 持续推送给前端。
5. 报告或分析结果入库并可导出。

### 4.2 定时任务流程

1. Celery Beat 按调度规则触发任务。
2. API/任务管理模块生成本次执行记录。
3. 调度 Agent 执行业务流程。
4. 报告完成后按照通知规则发送邮件或后续 IM 消息。

### 4.3 事件触发流程

1. 事件监控器持续轮询配置好的阈值或变更条件。
2. 条件满足后创建事件型任务。
3. 任务优先走“采集 → 分析 → 简报”最短路径。
4. 审阅要求以“速度优先但证据不能缺失”为准。

### 4.4 报告导出流程

1. 报告 Agent 输出标准 Markdown 结构。
2. Exporter 根据目标格式渲染 PDF / PPT。
3. 产物写入对象存储，并把元数据写回 PostgreSQL。
4. 前端提供下载入口与版本查看能力。

## 5. 接口与集成边界

### 5.1 内部接口分组

| 接口分组 | 主要职责 |
|------|------|
| Auth | 登录、刷新、登出、权限校验 |
| Users | 用户与角色管理 |
| DataSources | 数据源增删改查、测试、采集记录 |
| Tasks | 任务创建、执行、重试、状态查询 |
| Reports | 报告列表、详情、导出、版本 |
| Chat | 问答触发、会话历史、流式输出 |
| Templates | 模板管理 |
| Alerts / Notifications | 规则配置、记录查询 |

### 5.2 SSE 事件契约

| 事件 | 用途 | 必带字段 |
|------|------|------|
| `status` | 推送阶段状态 | phase, progress, message |
| `content` | 推送正文、图表、来源 | type, payload |
| `error` | 推送错误信息 | code, message, recoverable |
| `done` | 推送完成信号 | task_id, report_id |

### 5.3 外部集成边界

| 集成对象 | 方式 | 架构要求 |
|------|------|------|
| SearXNG | HTTP API | 仅作为搜索聚合入口，不承担业务逻辑 |
| 目标网站/媒体 | HTTP / 爬虫 | 通过采集适配器隔离站点差异 |
| Ollama | HTTP API | 模型服务可替换，业务层不直连具体模型名 |
| SMTP | 标准邮件协议 | 通知失败不得影响主任务完成态 |
| IM Webhook | Webhook | 作为通知扩展，不进入主路径强依赖 |

## 6. 横切约束

### 6.1 安全

- 所有外部入口均需鉴权
- 角色权限由后端强校验，前端只做展示裁剪
- 敏感配置通过环境变量或密钥管理注入
- 上传文件需做类型与大小校验

### 6.2 可观测性

- 每个任务必须具备唯一 `task_id`
- Agent 阶段日志必须可按 `task_id` 聚合
- 失败原因必须区分为采集失败、处理失败、模型失败、导出失败、通知失败
- 核心指标至少覆盖任务耗时、失败率、导出耗时、通知成功率

### 6.3 可靠性

- 长耗时任务必须支持超时控制
- 可恢复错误支持有限次重试
- 通知失败不应回滚已完成的报告
- 外部搜索或模型服务不可用时，应支持降级或快速失败

### 6.4 可扩展性

- 新数据源以适配器方式接入
- 新模型以统一推理接口接入
- 报告模板与 Prompt 模板必须解耦
- 后续拆分微服务时，以当前模块边界为拆分基础

## 7. 架构决策记录

| 决策 | 当前选择 | 原因 | 何时需要重审 |
|------|------|------|------|
| 架构形态 | Modular Monolith | MVP 交付速度更快、协调成本更低 | 团队规模扩大或吞吐显著上升 |
| 编排框架 | LangGraph | 适合状态化流程与多阶段编排 | 编排复杂度超出维护阈值 |
| 推理服务 | Ollama + 本地模型 | 降低外部依赖，便于私有部署 | 本地时延或效果长期不达标 |
| 搜索聚合 | SearXNG | 开源可自建、降低外部 API 依赖 | 搜索质量明显不足 |
| 对象存储 | MinIO | S3 兼容，便于自建部署 | 存储规模或运维成本超预期 |

### 7.2 架构变更触发条件

- PRD 新增核心能力或修改范围边界
- 开发实现证明当前模块边界不可维护
- 审阅发现重复返工来自架构职责不清
- 性能、稳定性或成本指标连续不达标
