## 1. 文档定位与使用方式

### 1.1 本文档回答的问题

本文件只回答以下问题：

- 为什么做这个系统
- 这个系统为谁服务
- MVP 必须交付什么能力
- 每项能力的业务验收标准是什么

本文件**不负责**描述具体技术方案、代码组织、依赖版本、实现顺序；这些内容分别由 `architecture.md`、`tech.md`、`plan.md` 维护。

### 1.2 多 Agent 使用约定

| Agent | 必读内容 | 可更新内容 | 不应直接更新的内容 |
|------|------|------|------|
| 设计 Agent | 全文，重点看第 3-6 节 | 需求边界、验收标准、风险假设 | 技术选型、代码路径、依赖版本 |
| 开发 Agent | 第 4-6 节 | 原则上不直接改 PRD；若实现暴露需求缺口，应先回写到设计结论，再由设计 Agent 更新 PRD | 技术方案替代 PRD |
| 审阅 Agent | 第 4-6 节 | 可补充“需求歧义/验收缺口” | 以实现细节覆盖业务目标 |

### 1.3 与其他核心文档的边界

| 文档 | 关注点 | 单一事实来源 |
|------|------|------|
| `prd.md` | 业务目标、范围、验收标准、非功能目标 | “做什么、做到什么算完成” |
| `architecture.md` | 系统边界、模块职责、关键流程、接口边界 | “系统如何分层与协作” |
| `tech.md` | 技术选型、版本、目录约定、工具链、依赖约束 | “具体采用什么技术实现” |
| `plan.md` | 任务拆分、依赖、状态流转、协作闭环 | “按什么顺序推进、当前进展如何” |

## 2. 产品背景与目标

### 2.1 产品愿景

构建一个面向市场、营销与管理团队的自动化市场洞察系统，形成“数据采集 → 数据处理 → AI 分析 → 报告交付”的闭环，显著降低人工信息收集成本并提升决策效率。

### 2.2 核心成功指标

| 目标 | 衡量指标 | 目标值 |
|------|------|------|
| 降低信息收集成本 | 人工收集时长下降比例 | ≥ 70% |
| 提升洞察产出速度 | 从任务创建到首版报告完成时间 | 标准任务 ≤ 2 小时 |
| 提升分析可用性 | 业务侧对报告“可直接用于决策”的认可率 | ≥ 80% |
| 提升数据覆盖度 | 核心数据源类型覆盖 | 新闻 / 社媒 / 竞品 / 上传文档 / 历史知识全覆盖 |
| 提升系统采用率 | 目标部门月活覆盖率 | ≥ 60% |

### 2.3 非目标（当前阶段不追求）

- 不在 MVP 阶段开放对外 API 平台
- 不在 MVP 阶段完成多租户架构
- 不在 MVP 阶段覆盖所有行业模板
- 不在 MVP 阶段交付复杂预测模型或因果推断平台

## 3. 用户与关键场景

### 3.1 角色定义

| 角色 | 核心职责 | 典型关注点 |
|------|------|------|
| 管理员（Admin） | 系统配置、权限管理、数据源配置、模板管理 | 系统稳定性、权限边界、配置正确性 |
| 分析师（Analyst） | 创建任务、发起问答、审阅与编辑报告 | 趋势、竞品、舆情、专题分析 |
| 查看者（Viewer） | 阅读报告、导出内容、跟踪结论 | 结论摘要、图表、行动建议 |

### 3.2 权限矩阵

| 功能 | Admin | Analyst | Viewer |
|------|------|------|------|
| 用户与权限管理 | 是 | 否 | 否 |
| 数据源配置 | 是 | 是 | 否 |
| 分析任务创建与执行 | 是 | 是 | 否 |
| 报告查看与导出 | 是 | 是 | 是 |
| 报告编辑与批注 | 是 | 是 | 否 |
| 通知规则配置 | 是 | 是 | 否 |
| 模板管理 | 是 | 是 | 否 |

### 3.3 核心业务场景

| 场景 | 触发方式 | 期望结果 |
|------|------|------|
| 专题分析 | 分析师手动发起 | 在限定时间范围内生成可追溯的专题报告 |
| 周期性周报/月报 | 定时任务 | 自动产出并分发固定结构报告 |
| 事件触发快报 | 指标或外部事件触发 | 在短时间内生成简报并通知相关人员 |
| 管理员配置新行业模板 | 后台维护 | 让分析师能直接复用模板发起任务 |

## 4. MVP 范围定义

### 4.1 MVP 包含内容（In Scope）

| 模块 | MVP 范围 | 业务价值 |
|------|------|------|
| 数据采集 | 搜索、爬虫、用户上传、历史知识检索 | 建立多源信息输入能力 |
| 数据处理 | 去重、清洗、标准化、语言识别、基础实体抽取 | 为分析提供可消费数据 |
| AI 分析 | 多 Agent 协作分析，输出结论、证据、置信度 | 形成结构化洞察 |
| 报告交付 | Markdown 报告、PDF 导出、PPT 导出 | 让结果可阅读、可传播 |
| 触发方式 | 问答触发、定时触发、事件触发 | 覆盖主动与被动使用场景 |
| 通知机制 | 邮件通知，预留 IM 集成 | 支撑报告与异常触达 |
| 权限与后台 | 基础 RBAC、数据源管理、模板管理 | 满足可控运行与日常运营 |

### 4.2 MVP 不包含内容（Out of Scope）

- 多租户隔离
- 对外开放 API
- 完整 GDPR 自动化流程
- 行业模板市场
- 高阶预测/归因/因果分析平台
- 全量多语言报告生产

### 4.3 阶段路线图

| 阶段 | 重点 | 结果 |
|------|------|------|
| Phase 1 | MVP 闭环 | 采集、处理、分析、报告、权限、通知 |
| Phase 2 | 增强分析 | 英文界面、更多模板、基础合规功能 |
| Phase 3 | 企业化集成 | 内部业务数据对接、开放能力、更多行业 |
| Phase 4 | 规模化 | 多租户、模板市场、更强自动化 |

## 5. 功能需求与验收标准

### 5.1 数据采集

| 编号 | 需求 | 验收标准 |
|------|------|------|
| FR-DC-01 | 支持搜索型外部信息采集 | 用户可基于主题、时间范围、关键词发起采集任务 |
| FR-DC-02 | 支持站点级爬虫采集 | 可配置目标站点并获取正文、标题、时间、来源 |
| FR-DC-03 | 支持用户上传文档 | 支持至少 PDF、Word、Excel 常见格式 |
| FR-DC-04 | 支持历史知识检索 | 历史报告或知识可作为上下文参与分析 |
| FR-DC-05 | 支持采集任务重试与状态查看 | 用户可看到 pending/running/success/failed 等状态 |

### 5.2 数据处理

| 编号 | 需求 | 验收标准 |
|------|------|------|
| FR-DP-01 | 支持跨来源去重 | 同一事件多来源内容不会在报告中重复堆叠 |
| FR-DP-02 | 支持字段标准化 | 时间、来源、语言、主题等关键字段结构统一 |
| FR-DP-03 | 支持语言识别与必要翻译 | 非中英文内容可被识别，必要时可转为统一输出语言 |
| FR-DP-04 | 支持基础实体抽取 | 至少识别公司、品牌、产品、人名、地区等实体 |

### 5.3 AI 分析

| 编号 | 需求 | 验收标准 |
|------|------|------|
| FR-AI-01 | 支持多 Agent 协作分析 | 采集、清洗、分析、报告各阶段职责清晰可追踪 |
| FR-AI-02 | 输出结论与证据 | 每个核心结论都能关联证据或来源 |
| FR-AI-03 | 输出置信度或不确定性提示 | 高风险结论必须标注不确定性 |
| FR-AI-04 | 支持多轮问答 | 用户可基于上一次结果继续追问 |

### 5.4 报告交付

| 编号 | 需求 | 验收标准 |
|------|------|------|
| FR-RP-01 | 支持结构化 Markdown 报告 | 报告包含摘要、发现、证据、图表或行动建议等章节 |
| FR-RP-02 | 支持 PDF 导出 | 导出结果排版可阅读、可分享 |
| FR-RP-03 | 支持 PPT 导出 | 能用于汇报场景，保留关键图表与结论 |
| FR-RP-04 | 支持模板驱动生成 | 同类任务可复用统一模板 |

### 5.5 触发与通知

| 编号 | 需求 | 验收标准 |
|------|------|------|
| FR-TG-01 | 支持问答触发 | 用户可通过对话框直接发起分析 |
| FR-TG-02 | 支持定时触发 | 任务可按日/周/月自动运行 |
| FR-TG-03 | 支持事件触发 | 达到阈值或检测到异常后自动生成简报 |
| FR-NT-01 | 支持邮件通知 | 报告完成与异常失败能通知相关人员 |

### 5.6 后台管理与权限

| 编号 | 需求 | 验收标准 |
|------|------|------|
| FR-AD-01 | 支持基础 RBAC | 至少区分 Admin / Analyst / Viewer |
| FR-AD-02 | 支持数据源管理 | 后台可增删改查并测试数据源 |
| FR-AD-03 | 支持模板管理 | 模板可维护、可选择、可版本化 |

## 6. 非功能需求

| 类别 | 要求 | 目标 |
|------|------|------|
| 性能 | 标准问答任务首个有效输出时间 | ≤ 60 秒 |
| 性能 | 标准报告完整生成时间 | ≤ 2 小时 |
| 可用性 | 核心服务可用性 | ≥ 99.5% |
| 安全 | 认证授权 | 必须有 JWT + RBAC |
| 安全 | 敏感数据保护 | 传输加密、关键数据加密存储 |
| 可观测性 | 全链路可追踪 | 任务、Agent、导出、通知均可查 |
| 可维护性 | 模块边界清晰 | 各模块可独立测试与替换 |
| 可扩展性 | 新数据源接入成本可控 | 通过适配器或插件方式新增 |

## 7. 风险与假设

### 7.1 关键风险

| 风险 | 影响 | 缓解思路 |
|------|------|------|
| 外部数据源变化或限流 | 采集能力不稳定 | 多源冗余、失败重试、降级策略 |
| 模型成本或时延超预期 | 交付速度与成本失控 | 多模型路由、任务分级、缓存 |
| 需求膨胀 | MVP 失焦 | 以第 4 节范围为准，变更必须先改 PRD |
| 报告可追溯性不足 | 审阅无法通过 | 将“结论-证据-来源”作为硬性验收项 |

### 7.2 关键假设

- 目标用户具备基础市场分析能力
- MVP 阶段主要服务中文场景，英文为次优先
- 团队可接受本地模型与开源组件部署成本
- 业务侧认可“先交付闭环、再增强深度分析”的推进方式

### 8.1 术语

| 术语 | 定义 |
|------|------|
| RBAC | 基于角色的访问控制 |
| RAG | 检索增强生成 |
| Agent | 具备明确职责、输入输出和状态边界的智能执行单元 |
| MVP | 最小可行产品 |

### 8.2 文档维护规则

- 需求变更先改 `prd.md`，再同步影响到 `architecture.md`、`tech.md`、`plan.md`
- 任何实现细节不得反向污染 PRD 的业务表述
- 审阅结论若指向“需求定义不清”，应回到 PRD 修订后再继续任务流转
